Exemplo Concreto

O contrato de retorno fragil

Os 4 chamadores precisam conhecer os bugs para funcionar

-- A procedure retorna sucesso assim: p_erro_controle := 'procedimento efetuado para empresa ,' || cd_empresa; -- O chamador extrai cd_empresa com parsing: l_emp := substr(w_erro, instr(w_erro, ',') + 1); -- Se for erro ORA-, o parsing retorna lixo: -- 'ORA-06502: PL/SQL: numeric or value error' -- l_emp = ' numeric or value error' -- A "defesa"? Checar se contem ORA-: if instr(l_emp, 'ORA-') > 0 then l_emp := null; end if;

Problemas

  • Sucesso e erro no mesmo parametro
  • Parsing com substr/instr (fragil)
  • Espaco antes da virgula inconsistente
  • Chamador precisa conhecer os bugs

Solucao proposta

  • fl_status = 'S'/'E'/'P' - inequivoco
  • cd_empresa - acesso direto
  • ds_etapa_erro - saber ONDE falhou
  • TYPE RECORD type-safe
02

O que e DDD?

Domain-Driven Design - em linguagem acessivel

DDD

Domain-Driven Design em um slide

Eric Evans, 2003 - usado por Netflix, Nubank, iFood, grandes bancos

Abordagem Tradicional

  • Comecar pelo banco de dados
  • Pensar em tabelas
  • Codigo tecnico que so dev entende
  • "Funciona"
  • Desenvolvedor decide sozinho

Abordagem DDD

  • Comecar pelo negocio
  • Pensar em processos
  • Codigo que fala a lingua do negocio
  • "Funciona E e claro"
  • Negocio + Dev decidem juntos

Por que DDD aqui? Dominio complexo (ANS, coparticipacao, carencia) | Regras intrincadas | Multiplos especialistas | Legado de 15+ anos | Evolucao continua necessaria

Decisoes de Modelagem

Decisoes arquiteturais-chave

Strategy Pattern (SIGO vs BITIX)

Dois cursors identicos (cr_empresa_neg e cr_empresa_neg_bitix) unificados via Strategy.

IOrigemPropostaStrategy
  -- SigoStrategy
  -- BitixStrategy

Ganho: elimina duplicacao; novos canais = nova Strategy.

Fail Fast (Correcao de Ordem)

Codigo da empresa gerado antes de validar = gaps na sequence.

Legado: Gerar codigo → Validar → Falhou? Codigo perdido
DDD: Validar → OK? → Gerar codigo → Zero desperdicio

Anti-Corruption Layer

Cursor legado (80+ colunas flat) → Objetos de dominio tipados via PropostaAdapter.

AFFIX como Policy

Contratos AFFIX possuem regras especiais (endereco, coparticipacao, limites). Modelado como Policy que modifica o comportamento.

Como sera feito

Roadmap em 4 fases - Strangler Fig Pattern

Roadmap

Strangler Fig Pattern

O sistema nunca para de funcionar. Cada fase e entregavel.

Fase 0 - Atual 4.991 linhas monoliticas
Fase 1 - Quick Wins ~4.200 linhas | Risco: MINIMO

Extrair log (-360 linhas) | Unificar cursors | Validacoes em package

Fase 2 - Packages DDD 18 packages | Risco: MEDIO

Cada BC → 1 package | Procedure vira orquestrador

Fase 3 - Orquestrador Limpo ~30 linhas legiveis | Risco: ALTO

Novo contrato de retorno | Adapter para compatibilidade

Fase 4 - Cloud / Azure Microservicos .NET | Risco: ALTO

ASP.NET Core | Azure Durable Functions (Saga) | Service Bus | Application Insights

Resultado - Fase 3

De 5.000 para ~30 linhas

Qualquer desenvolvedor novo entende em 5 minutos

PROCEDURE pr_cadastramento_empresa_prov( p_nu_controle IN NUMBER, p_resultado OUT pk_efetivacao_types.t_resultado_efetivacao ) AS v_ctx pk_cadastramento_empresa.t_contexto_cadastro; BEGIN v_ctx := pk_cadastramento_empresa.fn_inicializa_contexto(p_nu_controle); pk_validacao_proposta.pr_valida_completa(v_ctx); -- BC-02 pk_filial_area_venda.pr_resolve_filial(v_ctx); -- BC-05 pk_modelo_negocio.pr_resolve_parametros(v_ctx); -- BC-06 pk_pessoa_juridica.pr_cria_ou_atualiza(v_ctx); -- BC-03 pk_precificacao.pr_cria_tabelas_preco(v_ctx); -- BC-07 pk_empresa_conveniada.pr_cria_empresa(v_ctx); -- BC-08 pk_coparticipacao.pr_configura(v_ctx); -- BC-09 pk_carencia.pr_configura(v_ctx); -- BC-10 pk_endereco_comunicacao.pr_cadastra(v_ctx); -- BC-04 pk_acesso_internet.pr_configura(v_ctx); -- BC-12 pk_fidelizacao.pr_configura(v_ctx); -- BC-11 pk_reembolso.pr_configura(v_ctx); -- BC-15 pk_minimo_contratual.pr_configura(v_ctx); -- BC-16 COMMIT; pk_notificacao_email.pr_envia_efetivacao(v_ctx); -- BC-17 (pos-commit) pk_desconto_pim.pr_aplica_desconto(v_ctx); -- BC-18 pk_integracao_odonto.pr_integra(v_ctx); -- BC-13 p_resultado := pk_efetivacao_types.sucesso(v_ctx); EXCEPTION WHEN OTHERS THEN pk_cadastramento_empresa.pr_trata_erro(v_ctx, p_resultado); END;

Riscos e mitigacoes

Regressao funcional

Impacto Alto

Mitigacao: Testes de regressao comparando dados (47 tabelas) antes de cada fase. Feature flags para rollback imediato.

Boundary transacional

Impacto Alto

Mitigacao: COMMIT interno mantido ate Fase 3. Decisao documentada como ADR-03.

Resistencia do time

Impacto Medio

Mitigacao: Quick Wins visiveis na Fase 1 (resultado imediato). Treinamento DDD de 2 sessoes. 5 conceitos, nao 50.

Plano B (Rollback)

Sempre disponivel

Cada fase tem rollback independente. Procedure original coexiste como adapter. Feature flags para desligar packages em segundos.
